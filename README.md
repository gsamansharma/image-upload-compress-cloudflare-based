# Image Upload & Compression (Cloudflare Version)

A high-performance, production-ready image service powered by Cloudflare Workers and R2. This service automatically compresses uploads to WebP and converts them using WebAssembly, while providing advanced security through Domain Lockdown and HMAC-Signed URLs.

## 🚀 Features
- **Fast Uploads**: Global distribution via Cloudflare's edge network.
- **Wasm Compression**: Efficient image processing using WebAssembly.
- **Secure Persistence**: Images are stored in a private R2 bucket.
- **Hybrid URL Architecture**: 
  - **Permanent URLs**: Consistent identifiers for your database.
  - **Signed URLs**: Short-lived, secure access links for the frontend.
- **Security Hardening**:
  - **Domain Lockdown**: Restricted access to your custom domain.
  - **HMAC Signatures**: Prevents URL tampering and unauthorized viewing.
  - **API Key Auth**: Secured uploads via `X-API-Key`.
  - **Configurable Limits**: Set MAX_SIZE and Expiry via environment variables.

## 🛠 Setup

### 1. Project Initialization
```bash
# Install dependencies
npm install
```

### 2. Local Configuration (.dev.vars)
Create a `.dev.vars` file in the root directory. This is where you store local secrets (it is ignored by Git). Use the provided `.dev.vars.example` as a template:

```bash
AUTH_KEY=your_secret_key
SIGNING_SECRET=your_url_signing_secret
IMAGE_QUALITY=80
BASE_URL=https://images.example.com
ALLOWED_DOMAIN=images.example.com
R2_BINDING=BUCKET
MAX_SIZE_MB=10
URL_EXPIRY_MINUTES=60
```

### 3. Infrastructure setup
```bash
# Create R2 bucket
npx wrangler r2 bucket create image-uploads
```

### 4. Production Deployment
1. Set your secure secrets in Cloudflare:
   ```bash
   npx wrangler secret put AUTH_KEY
   npx wrangler secret put SIGNING_SECRET
   ```
2. Update your `wrangler.toml` with your account/zone details and deploy:
   ```bash
   npm run deploy
   ```

## 📡 API Usage

### POST /upload
Send a `multipart/form-data` request with an image file.
- **Header**: `X-API-Key: your_auth_key`
- **Field Name**: `image`
- **Response**:
    ```json
    {
      "message": "Image uploaded & compressed successfully!",
      "fileName": "unique-id.webp",
      "permanentUrl": "https://images.example.com/unique-id.webp",
      "previewUrl": "https://images.example.com/unique-id.webp?sig=...&exp=...",
      "originalSize": 1024,
      "compressedSize": 512
    }
    ```

### GET /:filename.webp
Viewing images requires a valid HMAC signature generated by your application using the `SIGNING_SECRET`.

## 🔒 Environment Variables Reference

| Variable | Description | Default |
| :--- | :--- | :--- |
| `AUTH_KEY` | Secret key for authorizing uploads. | (Required) |
| `SIGNING_SECRET` | Shared secret for generating HMAC signatures. | `dev-secret` |
| `IMAGE_QUALITY` | WebP compression quality (1-100). | `80` |
| `BASE_URL` | Your custom domain (e.g. `https://images.showmy.page`). | (Current Host) |
| `ALLOWED_DOMAIN` | The only domain permitted to serve images. | (All allowed) |
| `R2_BINDING` | The binding name for the R2 bucket in `wrangler.toml`. | `blob` |
| `MAX_SIZE_MB` | Maximum permitted file size for uploads. | `10` |
| `URL_EXPIRY_MINUTES` | Default duration for generated preview links. | `60` |

## 🔗 Architecture
For production, it is recommended to store the `fileName` or `permanentUrl` in your database. Your application (e.g., Next.js) should generate the `sig` and `exp` parameters on-the-fly when serving images to users.
